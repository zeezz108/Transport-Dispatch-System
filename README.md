# АвтоДиспетч — система автоматического вызова транспорта на основе прогноза отгрузок
Примечания (Файл CSV создается в корне проекта после обучения модели;
В папке data находятся тестовые данные)
## О проекте

**АвтоДиспетч** — это прототип системы автоматического вызова транспорта на склады, разработанный в рамках командного трека. Система решает ключевую логистическую проблему: преобразование прогнозных объёмов отгрузок в конкретные операционные решения — заявки на транспорт.

### Проблематика

Ошибки в планировании отгрузок и вызове транспорта приводят к:
- простоям складов и персонала
- задержкам отгрузки заказов
- росту операционных затрат
- неэффективному использованию транспортного парка

### Решение

АвтоДиспетч автоматизирует процесс:
1. **Прогнозирование** объёмов отгрузок на 20 часов вперёд (шаг = 2 часа)
2. **Расчёт необходимого количества транспорта**
3. **Формирование заявок** с учётом приоритетов и времени выполнения
4. **Предоставление API и веб-интерфейса** для интеграции и ручного управления

## Бизнес-логика работы

### Логика преобразования прогноза в заявки

1. **Прогноз**: модель предсказывает объём отгрузок `target_2h` на 10 шагов вперёд (20 часов)
2. **Расчёт машин**: `ceil(volume / TRUCK_CAPACITY)`, где `TRUCK_CAPACITY = 1.0` (одна машина = 1 единица объёма)
3. **Время вызова транспорта**: за `LEAD_TIME_HOURS = 4` часа до прибытия на склад
4. **Минимальный порог**: заявка создаётся только при `volume >= 1.0`

### Приоритизация заявок

| Шаг прогноза | Временной горизонт | Приоритет | Действие |
|-------------|-------------------|-----------|----------|
| 1-2 | 0-4 часа | **URGENT** (Срочный) | Немедленный вызов |
| 3-5 | 4-10 часов | **PLANNED** (Плановый) | Вызов по расписанию |
| 6-10 | 10-20 часов | **RESERVED** (Резерв) | Предварительное бронирование |

### Уверенность прогноза

| Шаг прогноза | Уровень уверенности | Использование |
|-------------|--------------------|---------------|
| 1-3 | HIGH (Высокая) | Можно полагаться для срочных решений |
| 4-6 | MEDIUM (Средняя) | Требует подтверждения |
| 7-10 | LOW (Низкая) | Только для предварительного планирования |

### Периодичность

- **Горизонт прогноза**: 10 шагов × 2 часа = 20 часов
- **Частота обновления**: каждые 2 часа (совпадает с временным шагом данных)
- **Лид-тайм вызова**: 4 часа (время на подачу транспорта)

## Прогнозная модель

### Архитектура HybridForecaster

Гибридный ансамбль из двух LightGBM моделей с контролем качества через сравнение с baseline:

- **Модель 1** (консервативная): вес 0.6, меньшая сложность
- **Модель 2** (агрессивная): вес 0.4, большая сложность
- **Ансамбль**: `prediction = 0.6 × m1 + 0.4 × m2`
- **Fallback**: скользящее среднее за 24 часа (если ансамбль не лучше baseline)

### Признаки для обучения

| Категория | Признаки |
|-----------|----------|
| Временные | час, день недели, день месяца, месяц, выходной (с циклическим кодированием sin/cos) |
| Бизнес-признаки | период выплат, начало/конец месяца, начало/конец выходных |
| Лаговые | lag_1, lag_2, lag_3, lag_4, lag_6, lag_8, lag_12, lag_24, lag_48 |
| Скользящие статистики | rolling_mean/std за 4, 8, 12, 24, 48 периодов |
| Трендовые | trend_7d, ratio_to_mean |
| Категориальные | office_from_id, route_id |
| Статусные (опц.) | total_in_process, weighted_status, has_high_status |

### Оценка качества модели

**Метрика**: MAE (Mean Absolute Error) — средняя абсолютная ошибка

**Контроль качества**: на каждом шаге прогноза сравниваем ансамбль с baseline (скользящее среднее за 24 часа):
- Если ансамбль лучше baseline → сохраняем ансамбль
- Если baseline лучше → используем baseline

## Бизнес-метрики оценки системы

### Основные KPI

| Метрика | Описание | Целевое значение |
|---------|----------|------------------|
| **MAE прогноза** | Средняя абсолютная ошибка предсказания объёма | < 1.5 ед. |
| **Точность приоритетов** | % заявок, у которых приоритет совпал с фактической срочностью | > 85% |
| **Простои складов** | Время ожидания транспорта после готовности отгрузки | < 30 мин |
| **Эффективность транспорта** | Средняя загрузка машин | > 70% |

### Тестирование и валидация

1. **Бэктестинг**: прогон исторических данных и сравнение с реальными отгрузками
2. **A/B тестирование**: сравнение работы системы с ручным планированием
3. **Бейзлайн**: сравнение с простыми правилами (например, "вызывать всегда N машин")

1. **Горизонтальное масштабирование**:

## Бизнес-допущения (MVP)

При разработке прототипа приняты следующие допущения:

| Допущение | Обоснование |
|-----------|-------------|
| `TRUCK_CAPACITY = 1.0` | Одна единица объёма отгрузки = один грузовик (упрощение для MVP) |
| `LEAD_TIME_HOURS = 4` | Транспорт подаётся за 4 часа до прибытия на склад |
| `PERIOD_HOURS = 2` | Прогноз и отгрузки агрегированы в 2-часовые интервалы |
| Все машины одинаковой вместимости | Игнорируем разный тоннаж для упрощения логики |
| Бесконечный парк транспорта | Нет ограничения на количество доступных машин |
| Время в пути фиксировано | Не учитываем пробки и задержки |
| Статусные признаки опциональны | Модель работает и без них (fallback) |
